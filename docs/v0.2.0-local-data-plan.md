# v0.2.0 本機資料版設計計畫

## 1. v0.2.0 本機資料版目標

v0.2.0 的目標是把目前展示用的 MockData 逐步替換成本機可讀寫資料，讓「生活維護管家」具備最小可用的本機資料能力。

本版只處理本機資料流：

- 建立、讀取、更新、刪除物品。
- 建立與讀取保養步驟卡。
- 建立與讀取週期排程。
- 由排程產生今日任務。
- 完成任務後建立保養維修紀錄。
- 保持高風險與未知風險不顯示 DIY 步驟。

本版不是雲端版，不處理登入、Firebase、通知、拍照或圖片管理。

## 2. 目前 MockData 使用位置

目前 MockData 是展示資料來源，使用位置如下：

- TodayScreen
  - 讀取 `MockData.tasks` 顯示今日任務。
  - 透過 `MockData.items` 對應任務的物品名稱。
  - 透過 `MockData.maintenanceCards` 對應任務的保養步驟卡與風險資訊。
- ItemsScreen
  - 讀取 `MockData.items` 顯示物品列表。
- HistoryScreen
  - 讀取 `MockData.maintenanceRecords` 顯示保養履歷。
  - 透過 `MockData.items` 對應紀錄的物品名稱與分類圖示。
- PreviewItemDropdown
  - 讀取 `MockData.items` 作為新增頁 preview 表單的物品選項。

## 3. 新增頁目前只是 preview

## 4. 核心資料關係

v0.2.0 仍沿用目前 PMS 鏈路：

```text
Item -> MaintenanceCard -> Schedule -> Task -> MaintenanceRecord
```

### Item

Item 是被管理的物品，例如冷氣、機車、租屋合約。

關鍵關係：

- `Item.id` 被 MaintenanceCard、Schedule、Task、MaintenanceRecord 的 `itemId` 參照。

### MaintenanceCard

MaintenanceCard 是保養步驟卡，描述某個物品的保養、檢查、到期提醒或維修紀錄流程。

關鍵關係：

- `MaintenanceCard.itemId` 指向 Item。
- `MaintenanceCard.steps` 保存低風險或中風險可顯示的靜態步驟。
- `MaintenanceCard.riskLevel` 決定是否能顯示 steps。
- `MaintenanceCard.safetyNotice` 保存安全提示。

### Schedule

Schedule 是週期排程，描述某張保養步驟卡何時應該產生任務。

關鍵關係：

- `Schedule.itemId` 指向 Item。
- `Schedule.cardId` 指向 MaintenanceCard。
- `Schedule.nextDueDate` 可用來產生下一筆 Task。

### Task

Task 是某一天要執行的任務。

關鍵關係：

- `Task.itemId` 指向 Item。
- `Task.cardId` 指向 MaintenanceCard。
- `Task.scheduleId` 指向 Schedule。
- 完成任務後可建立 MaintenanceRecord。

### MaintenanceRecord

MaintenanceRecord 是保養或維修完成後留下的履歷。

關鍵關係：

- `MaintenanceRecord.itemId` 指向 Item。
- `MaintenanceRecord.taskId` 可指向 Task。
- 用於 HistoryScreen 顯示物品履歷。

## 5. LocalMaintenanceRepository 建議介面

LocalMaintenanceRepository 是 UI 與本機資料之間的統一入口。

只做設計，不在本文件實作。

建議責任：

- 讀取全部 Item。
- 新增、更新、封存 Item。
- 讀取全部 MaintenanceCard。
- 依 `itemId` 讀取 MaintenanceCard。
- 新增、更新 MaintenanceCard。
- 讀取全部 Schedule。
- 新增、更新 Schedule。
- 讀取全部 Task。
- 依狀態或日期讀取 Task。
- 新增、更新 Task。
- 讀取全部 MaintenanceRecord。
- 依 `itemId` 讀取 MaintenanceRecord。
- 新增 MaintenanceRecord。

建議介面概念：

```text
getItems()
saveItem(item)
archiveItem(itemId)

getMaintenanceCards()
getMaintenanceCardsByItem(itemId)
saveMaintenanceCard(card)

getSchedules()
saveSchedule(schedule)

getTasks()
getTasksByDate(date)
saveTask(task)

getMaintenanceRecords()
getMaintenanceRecordsByItem(itemId)
saveMaintenanceRecord(record)
```

## 6. LocalStorageService 建議責任

LocalStorageService 負責本機資料的實際保存與讀取。

只做設計，不在本文件實作。

建議責任：

- 將資料序列化成本機格式。
- 從本機格式反序列化回 model。
- 處理首次啟動時的空資料。
- 處理本機資料版本欄位。
- 提供簡單備份點，避免寫入中斷造成資料遺失。

建議先用單一本機資料檔保存：

```text
items
maintenanceCards
schedules
tasks
maintenanceRecords
settings
schemaVersion
```

v0.2.0 不需要先做複雜資料庫抽象。等資料量、查詢條件與同步需求明確後，再決定是否改成 SQLite 或其他儲存方式。

## 7. MaintenanceTaskService 建議責任

MaintenanceTaskService 負責從排程產生任務，以及維護任務狀態。

只做設計，不在本文件實作。

建議責任：

- 根據 Schedule 的 `nextDueDate` 產生 Task。
- 避免同一個 schedule/date 重複產生 Task。
- 根據今天日期判斷 pending、overdue。
- 任務完成後更新 Task 狀態。
- 任務延期時更新 Task 狀態與日期。
- 完成任務後可協助計算 Schedule 下一次到期日。

v0.2.0 可以先採用簡單規則：

- App 啟動時檢查 Schedule。
- 若 `nextDueDate` 已到或在本週內，建立對應 Task。
- 只處理本機時間，不做通知。

### 固定週期與突發事項規則

固定週期任務必須保留原本排程基準，不可因逾期或補做而改變下一次週期。

固定週期規則：

- 日不移日。
- 週不移週。
- 月不移月。
- 季不移季。
- 半年不移半年。
- 年不移年。
- 排程基準不可因逾期或補做而改變。
- 逾期是逾期，補做是補做，下一次週期仍按原基準推算。

突發事項規則：

- 突然故障、突發事項屬於限期處理任務。
- 限期工程、限期修繕、突發維修完成後，必須建立 MaintenanceRecord，並進入 ItemHistory。
- 季、半年、年等較大週期任務，完成後也必須建立 MaintenanceRecord，並進入 ItemHistory。

民間版文案規則：

- 不使用 PMS。
- 不使用限一、限二。
- 不使用保養修理卡。
- 不使用史略卡。
- 使用「限期處理」。
- 使用「保養維修紀錄」。
- 使用「物品履歷」。

## 8. MaintenanceRecordService 建議責任

MaintenanceRecordService 負責把任務完成或維修紀錄轉成履歷。

只做設計，不在本文件實作。

建議責任：

- 完成 Task 後建立 MaintenanceRecord。
- 手動新增保養/維修紀錄時建立 MaintenanceRecord。
- 綁定 `itemId` 與可選的 `taskId`。
- 保存費用、店家、零件、照片路徑、備註等欄位。
- 提供 HistoryScreen 查詢需要的排序資料。

v0.2.0 可以先不處理圖片檔案，只保留 model 欄位與文字資料。

### 事件紀錄、結案與履歷規則

v0.2.0 的本機資料版應把週期保養、突發故障、限期處理、限期工程、限期修繕與更換零件，都抽象成一筆 MaintenanceRecord。

MaintenanceRecord 不一定一次填完。使用者可以在處理期間持續補資料，直到結案。

可持續補充的資料包含：

- 開始日期。
- 處理過程。
- 照片。
- 零件。
- 費用。
- 店家。
- 保固。
- 結果。
- 結案日期。

結案後，MaintenanceRecord 應自動進入 ItemHistory，供履歷頁整理成物品履歷。

使用者端文案規則：

- 使用「保養維修紀錄」。
- 使用「處理過程」。
- 使用「結案日期」。
- 使用「物品履歷」。
- 不使用「保養修理卡」。
- 不使用「史略卡」。
- 不使用「完工日期」。

高風險項目仍不得提供 DIY 維修步驟。系統只能記錄問題、照片、店家、費用、零件、保固與結果，並引導使用者尋求合格專業人員協助。

## 9. SafetyBoundaryService 安全規則

SafetyBoundaryService 只負責判斷與提示，不提供危險維修教學。

安全規則：

- `RiskLevel.low` 可以顯示靜態保養步驟。
- `RiskLevel.medium` 可以顯示靜態保養步驟，但應保留安全提示。
- `RiskLevel.high` 不顯示 DIY steps。
- `RiskLevel.unknown` 不顯示 DIY steps。
- unknown 一律當高風險處理。
- 高風險項目只能提示使用者尋求合格專業人員協助。
- 高風險項目只能記錄問題、照片、店家、費用、零件、保固與後續追蹤。

不可產生或顯示下列類型的 DIY 維修步驟：

- 電力。
- 瓦斯。
- 煞車。
- 冷媒。
- 結構施工。
- 高壓設備。
- 高溫設備。
- 大型機械。
- 承重牆、樑柱。
- 醫療診斷或用藥調整。

## 12. high / unknown risk 不顯示 steps 的紅線

這是 v0.2.0 不能破壞的安全紅線。

必須維持：

- `RiskLevel.high` 不顯示 DIY steps。
- `RiskLevel.unknown` 不顯示 DIY steps。
- `RiskLevel.unknown` 一律當高風險處理。
- `SafetyBoundaryService` 只能做判斷與提示。
- `SafetyBoundaryService` 不可提供危險維修教學。
- 不可產生電力、瓦斯、煞車、冷媒、結構施工、高壓設備等 DIY 維修步驟。

建議 UI 判斷規則：

```text
if riskLevel == high or riskLevel == unknown:
  hide steps
  show safety notice
  guide user to qualified professional
else:
  show static maintenance steps
```

目前保養步驟卡 preview 已採用這條規則；v0.2.0 接本機資料後仍必須沿用同一條安全判斷。
