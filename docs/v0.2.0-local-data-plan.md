# v0.2.0 本機資料版設計計畫

## 1. v0.2.0 本機資料版目標

v0.2.0 的目標是把目前展示用的 MockData 逐步替換成本機可讀寫資料，讓「生活維護管家」具備最小可用的本機資料能力。

本版只處理本機資料流：

- 建立、讀取、更新、刪除物品。
- 建立與讀取保養步驟卡。
- 建立與讀取週期排程。
- 由排程產生今日任務。
- 完成任務後建立保養維修紀錄。
- 保持高風險與未知風險不顯示 DIY 步驟。

本版不是雲端版，不處理登入、Firebase、通知、拍照或圖片管理。

目前 main 已完成的本機資料能力：

- `ItemLocalRepository` 已完成。
- 新增物品已可儲存到本機。
- ItemsScreen 已可讀取本機物品；本機沒有資料時仍使用 MockData 顯示展示資料。
- `MaintenanceRecordLocalRepository` 已完成。
- 保養維修紀錄 preview 已可儲存到本機。
- HistoryScreen 已可讀取本機 MaintenanceRecord；本機沒有資料時仍使用 MockData 顯示展示資料。
- PreviewItemDropdown 已可優先讀取本機 items；本機沒有資料時 fallback 到 MockData。
- HistoryScreen 已可讀取本機 items，並用本機 items 對應本機紀錄的物品名稱與分類圖示。
- DemoDataNotice 已更新為「本機儲存 / 尚未支援雲端同步」的狀態說明。
- `ScheduleLocalRepository` 已完成。
- `TaskLocalRepository` 已完成。
- 新增到期提醒已可儲存 Schedule 到本機。
- TodayScreen 已可讀取本機 Schedule 與 Task。
- TodayScreen 已可由到期 Schedule 產生 due Task，並存回本機 Task 清單。
- TodayScreen 已可優先顯示本機 item 名稱；本機 items 找不到時 fallback 到 MockData items。
- manual-expiry-reminder 任務不會誤開保養步驟卡，只顯示到期提醒提示。
- TodayScreen 完成按鈕已可將本機 Task 更新為 completed。
- 完成本機 Task 後，會自動建立最小 MaintenanceRecord，供 HistoryScreen 顯示。

目前 main 仍未完成：

- Firebase、登入、雲端同步、通知、照片與 AI 仍未實作。

## v0.4.0 保養步驟執行版目前閉環

v0.4.0 名稱為保養步驟執行版。本版不是教使用者維修，而是讓低風險保養步驟可以被逐步勾選，並銜接既有的本機任務完成與保養維修紀錄流程。

v0.4.0 最小閉環：

```text
low risk 保養步驟卡 -> checklist -> 使用者逐步勾選 steps -> 全部 steps 完成 -> preview sheet callback -> TodayScreen _completeTask() -> Task completed -> MaintenanceRecord -> HistoryScreen
```

已完成的主線：

- low risk 保養步驟卡顯示 checklist。
- 使用者可在 preview sheet 內逐步勾選 steps。
- 所有 steps 完成後，才顯示可執行的完成流程。
- preview sheet 透過 `onCompleteSteps` callback 通知 TodayScreen。
- TodayScreen 重用既有 `_completeTask()`。
- Task 標記為 completed。
- 完成 Task 後建立 MaintenanceRecord。
- HistoryScreen 可讀取本機 MaintenanceRecord 並顯示紀錄。

### v0.4.0 安全邊界

v0.4.0 必須維持：

## 4. 核心資料關係

v0.2.0 仍沿用目前核心資料鏈路：

```text
Item -> MaintenanceCard -> Schedule -> Task -> MaintenanceRecord
```

### Item

Item 是被管理的物品，例如冷氣、機車、租屋合約。

關鍵關係：

- `Item.id` 被 MaintenanceCard、Schedule、Task、MaintenanceRecord 的 `itemId` 參照。

### MaintenanceCard

MaintenanceCard 是保養步驟卡，描述某個物品的保養、檢查、到期提醒或維修紀錄流程。

關鍵關係：

- `MaintenanceCard.itemId` 指向 Item。
- `MaintenanceCard.steps` 保存低風險或中風險可顯示的靜態步驟。
- `MaintenanceCard.riskLevel` 決定是否能顯示 steps。
- `MaintenanceCard.safetyNotice` 保存安全提示。

### Schedule

Schedule 是週期排程，描述某張保養步驟卡何時應該產生任務。

關鍵關係：

- `Schedule.itemId` 指向 Item。
- `Schedule.cardId` 指向 MaintenanceCard。
- `Schedule.nextDueDate` 可用來產生下一筆 Task。

### Task

Task 是某一天要執行的任務。

關鍵關係：

- `Task.itemId` 指向 Item。
- `Task.cardId` 指向 MaintenanceCard。
- `Task.scheduleId` 指向 Schedule。
- 完成任務後可建立 MaintenanceRecord。

### MaintenanceRecord

MaintenanceRecord 是保養或維修完成後留下的履歷。

關鍵關係：

- `MaintenanceRecord.itemId` 指向 Item。
- `MaintenanceRecord.taskId` 可指向 Task。
- 用於 HistoryScreen 顯示物品履歷。

## 7. MaintenanceTaskService 建議責任

MaintenanceTaskService 負責從排程產生任務，以及維護任務狀態。

只做設計，不在本文件實作。

建議責任：

- 根據 Schedule 的 `nextDueDate` 產生 Task。
- 避免同一個 schedule/date 重複產生 Task。
- 根據今天日期判斷 pending、overdue。
- 任務完成後更新 Task 狀態。
- 任務延期時更新 Task 狀態與日期。
- 完成任務後可協助計算 Schedule 下一次到期日。

目前 main 已完成的簡單規則：

- TodayScreen 進入時檢查本機 Schedule。
- 若 `nextDueDate` 已到期，建立對應 Task。
- 同一個 `scheduleId` 與同一天 `dueDate` 不重複建立 Task。
- 產生的 Task 會儲存到本機 Task 清單。
- TodayScreen 顯示更新後的本機 Task。
- 只處理本機時間，不做通知。

### 固定週期與突發事項規則

固定週期任務必須保留原本排程基準，不可因逾期或補做而改變下一次週期。

固定週期規則：

- 日不移日。
- 週不移週。
- 月不移月。
- 季不移季。
- 半年不移半年。
- 年不移年。
- 排程基準不可因逾期或補做而改變。
- 逾期是逾期，補做是補做，下一次週期仍按原基準推算。

突發事項規則：

- 突然故障、突發事項屬於限期處理任務。
- 限期工程、限期修繕、突發維修完成後，必須建立 MaintenanceRecord，並進入 ItemHistory。
- 季、半年、年等較大週期任務，完成後也必須建立 MaintenanceRecord，並進入 ItemHistory。

民間版文案規則：

- 不使用保養修理卡。
- 不使用軍中術語。
- 使用「限期處理」。
- 使用「保養維修紀錄」。
- 使用「物品履歷」。

### 事件紀錄、結案與履歷規則

v0.2.0 的本機資料版應把週期保養、突發故障、限期處理、限期工程、限期修繕與更換零件，都抽象成一筆 MaintenanceRecord。

MaintenanceRecord 不一定一次填完。使用者可以在處理期間持續補資料，直到結案。

可持續補充的資料包含：

- 開始日期。
- 處理過程。
- 照片。
- 零件。
- 費用。
- 店家。
- 保固。
- 結果。
- 結案日期。

結案後，MaintenanceRecord 應自動進入 ItemHistory，供履歷頁整理成物品履歷。

使用者端文案規則：

- 使用「保養維修紀錄」。
- 使用「處理過程」。
- 使用「結案日期」。
- 使用「物品履歷」。
- 不使用「保養修理卡」。
- 不使用「完工日期」。
- 不使用軍中術語。

高風險項目仍不得提供 DIY 維修步驟。系統只能記錄問題、照片、店家、費用、零件、保固與結果，並引導使用者尋求合格專業人員協助。

## 9. SafetyBoundaryService 安全規則

SafetyBoundaryService 只負責判斷與提示，不提供危險維修教學。

安全規則：

- `RiskLevel.low` 可以顯示靜態保養步驟。
- `RiskLevel.medium` 可以顯示靜態保養步驟，但應保留安全提示。
- `RiskLevel.high` 不顯示 DIY steps。
- `RiskLevel.unknown` 不顯示 DIY steps。
- unknown 一律當高風險處理。
- 高風險項目只能提示使用者尋求合格專業人員協助。
- 高風險項目只能記錄問題、照片、店家、費用、零件、保固與後續追蹤。

不可產生或顯示下列類型的 DIY 維修步驟：

- 電力。
- 瓦斯。
- 煞車。
- 冷媒。
- 結構施工。
- 高壓設備。
