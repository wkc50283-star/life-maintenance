# 生活維護管家 App 產品級總架構

## 1. App 核心定位

生活維護管家不是單純的 Todo App，而是「生活維護履歷系統」。

它的核心任務是協助使用者把日常物品、保養提醒、維修紀錄、費用、保固與後續追蹤整理成可長期保存的物品履歷。

使用者端一律使用民間語彙，例如：

- 生活維護
- 物品履歷
- 保養步驟卡
- 保養維修紀錄
- 到期提醒

產品內不對使用者顯示 PMS、MR 卡、史略卡等軍中術語。

## 2. 一句話主軸

買了東西拍一下，App 幫你提醒保養、記錄維修、留下完整履歷。

更濃縮的產品主軸：

一個東西，一張照片，一組提醒，一份紀錄，一生履歷。

## 4. 核心資料模型

### User 使用者

第一版可以先不登入，使用者資料先保留在本機設定中。

```text
User
├─ id
├─ name
├─ email
├─ createdAt
├─ reminderTime
└─ settings
```

### Item 物品

使用者端稱為「物品」。內部若需要抽象命名，可視為被維護的目標。

#### Item 與維護對象用語

程式內部、資料模型、repository、service 與資料格式仍維持使用 `Item`，本次文件更新不改 model、repository、service 或既有資料格式。

產品語言與使用者介面可以逐步把「物品」升級為「維護對象」。維護對象的範圍可以包含設備、車輛、房屋部位、文件、保險、健康檢查、長輩照護、寵物照護等，需要被提醒、紀錄、追蹤與形成履歷的對象。

未來若要調整 UI 文案，應採取最小修改原則，每次只替換明確範圍內的顯示文字，不同時改資料模型、儲存格式或流程邏輯。

維護對象的範圍擴大，不代表安全紅線放寬。電力、瓦斯、煞車、冷媒、結構施工、高壓設備、高溫設備，以及醫療診斷或用藥調整等高風險或未知風險項目，仍不得提供 DIY 維修或處置步驟，只能記錄問題、店家、費用、結果與後續追蹤，並引導尋求合格專業人員。

```text
Item
├─ id
├─ name
├─ category
├─ photoPath
├─ createdAt
├─ purchaseDate
├─ warrantyEndDate
├─ expectedLifeYears
├─ location
├─ note
└─ status
```

範例：

```text
客廳冷氣
類別：家電
建立日：2026/07/03
管理年限：5 年
保固到期：2029/07/03
```

### MaintenanceCard 保養步驟卡

保養步驟卡描述任務內容、週期、風險、步驟與完成時需要的紀錄。

```text
MaintenanceCard
├─ id
├─ itemId
├─ title
├─ taskType
├─ riskLevel
├─ cycleType
├─ estimatedMinutes
├─ steps
├─ requiredPhotos
├─ requiredNote
└─ safetyNotice
```

任務類型：

- 清潔
- 檢查
- 更換
- 維修紀錄
- 到期提醒
- 施工紀錄

風險等級：

- 低風險
- 高風險
- 不確定

### Step 步驟

```text
Step
├─ id
├─ cardId
├─ order
├─ title
├─ description
├─ isRequired
├─ photoRequired
└─ noteRequired
```

範例：

```text
1. 關閉電源
2. 拍清潔前照片
3. 取出濾網
4. 清洗濾網
5. 晾乾
6. 裝回
7. 測試運轉
```

### Schedule 提醒週期

```text
Schedule
├─ id
├─ itemId
├─ cardId
├─ cycleType
├─ interval
├─ startDate
├─ nextDueDate
├─ reminderTime
├─ enabled
└─ strictPeriodMode
```

週期包含：每日、每週、每月、每季、每半年、每年、自訂。

### Task 到期任務

```text
Task
├─ id
├─ itemId
├─ cardId
├─ scheduleId
├─ title
├─ dueDate
├─ status
├─ completedAt
├─ postponedAt
└─ overdue
```

狀態包含：待處理、已完成、已逾期、稍後提醒、取消。

### 固定週期與突發事項規則

固定週期任務必須保留原本排程基準，不可因逾期或補做而改變下一次週期。

固定週期規則：

- 日不移日。
- 週不移週。
- 月不移月。
- 季不移季。
- 半年不移半年。
- 年不移年。
- 排程基準不可因逾期或補做而改變。
- 逾期是逾期，補做是補做，下一次週期仍按原基準推算。

突發事項規則：

- 突然故障、突發事項屬於限期處理任務。
- 限期工程、限期修繕、突發維修完成後，必須建立保養維修紀錄，並進入物品履歷。
- 季、半年、年等較大週期任務，完成後也必須建立保養維修紀錄，並進入物品履歷。

民間版文案規則：

- 不使用 PMS。
- 不使用限一、限二。
- 不使用保養修理卡。
- 不使用史略卡。
- 使用「限期處理」。
- 使用「保養維修紀錄」。
- 使用「物品履歷」。

### MaintenanceRecord 保養維修紀錄

保養維修紀錄保存實際發生的維護、故障、維修、更換零件、費用與店家資訊。

```text
MaintenanceRecord
├─ id
├─ itemId
├─ taskId
├─ recordType
├─ date
├─ title
├─ issueDescription
├─ workDescription
├─ partsChanged
├─ cost
├─ vendorName
├─ warrantyUntil
├─ result
├─ photos
├─ note
└─ createdAt
```

紀錄類型：

- 定期保養
- 故障
- 維修
- 更換零件
- 到期處理
- 施工
- 其他

### 事件紀錄、結案與履歷規則

週期保養、突發故障、限期處理、限期工程、限期修繕與更換零件，都應抽象成一筆 MaintenanceRecord。這筆保養維修紀錄可以在處理期間持續補資料，直到使用者結案。

MaintenanceRecord 可記錄：

- 開始日期。
- 處理過程。
- 照片。
- 零件。
- 費用。
- 店家。
- 保固。
- 結果。
- 結案日期。

結案後，系統應自動將該保養維修紀錄整理進 ItemHistory，並在使用者端呈現為物品履歷。

使用者端文案規則：

- 使用「保養維修紀錄」。
- 使用「處理過程」。
- 使用「結案日期」。
- 使用「物品履歷」。
- 不使用「保養修理卡」。
- 不使用「史略卡」。
- 不使用「完工日期」。

高風險項目仍不得提供 DIY 維修步驟，只能記錄問題、照片、店家、費用、零件、保固與結果。

### ItemHistory 物品履歷

物品履歷不讓使用者手動填寫，而是由系統從 Item、Task、MaintenanceRecord 自動整理。

```text
ItemHistory
├─ itemId
├─ itemCreatedAt
├─ timeline
├─ totalCost
├─ lastMaintenanceDate
├─ nextDueDate
├─ partsHistory
└─ vendorHistory
```

畫面呈現範例：

```text
客廳冷氣履歷

2026/07/03
建立物品

2026/08/03
完成定期保養：清洗濾網
照片：有
結果：正常

2026/12/10
故障維修：冷氣不冷
店家：XX 冷氣行
費用：1800 元
更換零件：無
結果：恢復正常
```

## 5. 保養步驟卡生成架構

產品不應為每個物品手寫大量固定卡片，而是使用「骨架生成」的方式產生保養步驟卡。

### 6 種基本卡片骨架

1. 清潔型
2. 檢查型
3. 更換型
4. 維修紀錄型
5. 到期提醒型
6. 施工紀錄型

### 生成公式

```text
物品類型
+ 任務類型
+ 週期
+ 風險等級
+ 是否需要照片
+ 是否需要費用 / 零件紀錄
= 保養步驟卡
```

範例：

```text
冷氣 + 清潔 + 每月 + 低風險
= 冷氣濾網清洗步驟卡

冷氣 + 冷媒維修 + 高風險
= 專業維修紀錄卡，不提供 DIY 維修步驟
```

## 6. 安全界線系統

安全界線是產品底層規則。系統必須依風險等級決定能提供的內容。

```text
SafetyBoundary
│
├─ LowRisk
│  ├─ 可提供步驟
│  ├─ 可提醒
│  ├─ 可拍照紀錄
│  └─ 可完成打勾
│
├─ HighRisk
│  ├─ 不提供 DIY 維修教學
│  ├─ 顯示安全提醒
│  ├─ 引導找合格專業人員
│  ├─ 只記錄問題 / 照片 / 店家 / 費用 / 零件 / 保固 / 後續追蹤
│  └─ 寫入物品履歷
│
└─ UnknownRisk
   └─ 一律當高風險處理
```

### 高風險類別

- 電力
- 瓦斯
- 煞車
- 冷媒
- 結構施工
- 高壓設備
- 高溫設備
- 大型機械
- 承重牆 / 樑柱
- 醫療診斷 / 用藥調整

### 健康 / 老人照顧安全紅線

健康與老人照顧模組只能做提醒與紀錄：

- 不做診斷
- 不給醫療建議
- 不調整藥量
- 異常情況引導尋求醫師或合格醫療人員

## 7. 主要使用流程

### 流程 1：新增物品

```text
打開 App
→ 按「新增物品」
→ 拍照
→ 輸入名稱
→ 選分類
→ 選管理年限
→ 選保固日期
→ 勾選提醒週期
→ 產生保養步驟卡
→ 排入提醒
```

### 流程 2：到期提醒

```text
手機通知
→ 點進 App
→ 顯示今日任務
→ 查看保養步驟卡
→ 照步驟完成
→ 填寫保養維修紀錄
→ 自動寫入物品履歷
```

### 流程 3：東西壞了

```text
選物品
→ 新增問題
→ 選「故障 / 維修」
→ 拍照
→ 描述問題
→ 系統判斷風險
→ 低風險：可提供檢查步驟
→ 高風險：引導找合格專業人員
→ 維修後填寫紀錄
→ 自動寫入物品履歷
```

### 流程 4：換零件

```text
選物品
→ 新增紀錄
→ 類型：更換零件
→ 填零件名稱
→ 填日期
→ 填費用
→ 拍照
→ 設定下次提醒
→ 寫入物品履歷
```

### 流程 5：施工項目

```text
新增工作流程
→ 選施工 / 修繕類型
→ 套用流程模板
→ 每一步打勾
→ 每一步可拍照 / 備註
→ 完成後建立施工紀錄
→ 若綁定物品 / 房屋，寫入履歷
```

### 流程 6：老人照顧 / 健康檢查（後期）

```text
新增照顧對象
→ 建立健康檢查提醒
→ 建立回診提醒
→ 建立照顧紀錄
→ 完成後寫入照顧履歷
```

- 高風險情境有明確安全界線

## 11. 產品主軸總結

生活維護管家是一套把「物品、提醒、保養、維修、費用、保固、照片與後續追蹤」串成履歷的生活維護系統。

它的產品化說法是：

買了東西拍一下，該保養時提醒你；修過什麼、換過什麼、花多少錢，全部自動整理成履歷。

後續的老人照顧、健康檢查、施工 SOP、家庭共享、企業維保，都應從這個核心延伸，而不是取代核心。
