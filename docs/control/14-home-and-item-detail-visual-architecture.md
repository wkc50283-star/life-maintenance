# 生活管理 App 首頁與生活項目詳情資訊架構／視覺樣板規格

狀態：正式控制文件候選  
版本：v0.5.0  
日期：2026-07-18

## 1. 目的

在 schema v2 與正式 UI 施工前，先確認使用者是否能直接看懂：

- 現在最需要注意什麼
- 哪些事情只是提醒
- 哪些事情已經進入處理案件
- 哪些階段性重點或大修正在接近
- 某一生活項目長期需要管理哪些內容
- 完成後的過程與結果在哪裡查看

本文件只鎖定資訊架構、視覺層級與樣板驗收，不直接修改正式資料來源或既有 UI。

## 2. 現況稽核

### 2.1 首頁現況

目前第一分頁名稱為「今日」，內容主要是：

- 今日摘要 Hero
- 待處理任務清單
- 任務完成與建立舊 MaintenanceRecord

問題：

- 首頁被縮成任務清單，無法代表完整生活管理。
- 沒有進行中案件、階段性重點、大修或近期變化。
- 使用者不能一眼理解提醒、案件與史略的差別。
- AppBar、Hero、卡片與底部導覽視覺層級接近，缺乏主畫面焦點。

### 2.2 生活項目詳情現況

目前生活項目詳情使用大型 Bottom Sheet，內容依序是：

- 基本欄位表格
- 提醒事項
- 保養安排
- 處理紀錄

問題：

- 長內容塞在 Bottom Sheet，不適合長期管理頁。
- 基本資料佔據最前方，真正重要的狀態被壓到下方。
- 沒有 MaintenancePlan、Milestone、WorkCase、WorkCaseClosure 的位置。
- 「處理紀錄」仍偏向舊 MaintenanceRecord，不能呈現完整案件時間軸。
- 缺乏可辨識的視覺主體與情境感。

## 3. 首頁正式定位

首頁不是一般待辦清單，也不是只顯示今日任務。

首頁回答：

> 我現在的生活管理，有哪些事情需要看一眼？

首頁第一版資訊層級：

```text
首頁
├─ 今日概況
├─ 現在需要處理
├─ 進行中的案件
├─ 即將到來的階段性重點／大修
└─ 最近完成
```

### 3.1 今日概況

只顯示少量摘要：

- 今日日期與簡短問候
- 今日待處理數
- 進行中案件數
- 近期階段性重點數

不得使用完成率、連續打卡、紅色責備或評分。

### 3.2 現在需要處理

整合：

- 今日到期提醒
- 已逾期但尚未處理的提醒
- 手動建立的突發事項入口

每張卡只顯示：

- 生活項目
- 事項名稱
- 日期／狀態
- 下一個最直接動作

不得在卡片內塞入完整表單。

### 3.3 進行中的案件

顯示尚未結案的 WorkCase，例如：

- 等待廠商回覆
- 等待零件
- 施工中
- 補件中

每張卡至少顯示：

- 案件名稱
- 所屬生活項目
- 目前狀態
- 最後更新時間
- 下一步

### 3.4 階段性重點／大修

顯示已達標或即將達標的 Milestone，例如：

- 第六年大修
- 兩萬公里大保養
- 證件換發
- 照護階段轉換

首頁不顯示全部規則，只顯示最需要注意的少量項目。

### 3.5 最近完成

顯示最近結案或完成的少量紀錄，提供「已經處理好了」的安心感。

不得把此區做成績效排行榜。

## 4. 生活項目詳情正式定位

生活項目詳情應改為完整頁面，不再以大型 Bottom Sheet 作為正式長期入口。

它回答：

> 這個生活項目現在是什麼狀態？長期要管理什麼？過去發生過什麼？

正式資訊層級：

```text
生活項目詳情
├─ 項目主視覺與目前狀態
├─ 需要注意
├─ 保養項目
├─ 提醒與排程
├─ 階段性重點／大修
├─ 進行中案件
├─ 史略
└─ 基本資料
```

### 4.1 項目主視覺與目前狀態

首屏顯示：

- 項目名稱
- 系統大類與自訂類別
- 位置或關係描述
- 目前狀態
- 一個代表性圖示或照片區

不得把全部資料列在首屏。

### 4.2 需要注意

依優先順序顯示：

1. 進行中案件
2. 已達標 Milestone
3. 即將到期提醒
4. 暫停或異常資料

若沒有事項，顯示平靜的正常狀態，不製造焦慮。

### 4.3 保養項目

顯示 MaintenancePlan，不直接顯示模板目錄。

每張保養項目卡顯示：

- 名稱
- 風險層級
- 週期摘要
- 下次日期
- 狀態

### 4.4 提醒與排程

一般提醒與保養排程可分組顯示，但不得再依 `cardId` 猜測角色。

畫面名稱與資料角色必須一致：

- 保養項目
- 一般提醒
- 階段性重點
- 手動事項

### 4.5 階段性重點／大修

顯示：

- 重點名稱
- 觸發條件
- 目前進度或目標日期
- 狀態
- 是否已建立案件

### 4.6 進行中案件

顯示尚未結案的 WorkCase，點入後進入案件時間軸頁。

### 4.7 史略

史略是統一歷史視圖，可包含：

- 舊 MaintenanceRecord
- WorkCase 結案摘要
- WorkCaseUpdate 時間軸
- 任務完成資料
- 費用、廠商、零件與附件

不得把舊紀錄偽造成不存在的案件過程。

### 4.8 基本資料

基本資料放在較後段或次頁，包含：

- 建立日期
- 購買日期
- 保固日期
- 管理年限
- 備註
- 封存狀態

## 5. 導覽原則

第一階段保留五個底部入口，不在本文件直接批准改名：

- 今日／首頁
- 我的項目
- 新增
- 履歷
- 設定

但視覺樣板必須比較：

- 第一分頁使用「首頁」或「今日」哪一個更符合完整生活管理。
- 「履歷」是否維持全域入口，或改為「史略」。

名稱變更必須經畫面樣板與使用者批准，不得由開發者自行決定。

## 6. 視覺方向

正式方向：

> 溫暖、可信、成熟、有生活感，但不是醫療系統、工程後台或制式待辦工具。

### 6.1 應保留

- 柔和暖底色
- 清楚的大字與足夠留白
- Material 3 的可及性基礎
- 資料保護警告的高辨識度

### 6.2 必須改善

- 不再所有卡片都使用相同白底、灰框與圓角。
- 首頁要有明確主視覺層級。
- 重要狀態使用圖示、色階與版面位置共同表達，不只靠文字標籤。
- 卡片密度需分級：摘要卡、事件卡、歷史卡不能長得一樣。
- 禁止大量小膠囊、假按鈕與無功能裝飾。
- 不使用刺眼紅色製造未完成焦慮；紅色只用於真正危險或資料錯誤。

### 6.3 樣板尺寸

至少驗收：

- iPhone 13／14 類尺寸
- 小尺寸 iPhone
- 大字體模式
- 空資料
- 少量資料
- 多筆案件與歷史資料
- 本機資料保護警告存在時

## 7. 視覺樣板批次邊界

下一個視覺批次只允許建立：

1. 首頁靜態樣板
2. 生活項目詳情靜態樣板
3. 必要的展示資料模型或 preview fixture
4. Widget test／golden test 基礎（若專案環境可穩定重現）

明確禁止：

- 不接正式 Repository
- 不切換 Drift
- 不遷移資料
- 不修改現有任務完成流程
- 不刪除現有 TodayScreen 或 ItemDetailSheet
- 不把樣板入口放進正式底部導覽
- 不宣稱樣板就是正式完成 UI

## 8. 樣板驗收問題

使用者批准前必須逐項回答：

- 首頁是否一眼看懂現在需要注意什麼？
- 提醒、案件、階段性重點是否容易區分？
- 是否仍像一般待辦清單？
- 生活項目詳情是否能看懂長期管理結構？
- 保養項目與保養／修理案件是否被清楚分開？
- 大修是否有合理位置？
- 史略是否容易找到？
- 視覺是否有正式產品感？
- 字是否夠大、操作是否直覺？
- 使用者是否願意每天打開，而不是想立即刪除？

## 9. 批准條件

只有在兩張手機畫面樣板由專案負責人明確批准後，才可：

- 將資訊架構轉為正式 UI 施工計畫
- 決定第一分頁正式名稱
- 決定詳情頁導覽與區塊順序
- 調整全域 Theme
- 解除 schema v2 的下一階段設計阻擋

CI 通過不等於視覺批准。
