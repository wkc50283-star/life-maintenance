# 生活管理 App 開發與 Codex 規範

狀態：正式控制文件  
適用對象：人工開發者、ChatGPT、Codex 與其他自動化工具。

## 1. 不得混淆專案

- `life-maintenance` 是生活管理 App。
- PMS 是另一個產品概念，不得把 PMS 憲法、文案或資料角色直接帶入本 repo。
- 需求不明時先判斷專案身分，不得自行融合。

## 2. 最小修改

- 每次只修改完成該任務所需的最少檔案。
- 同一檔案只改已確認有問題的最小區塊。
- 不為了排版、重構或「更乾淨」而順便重寫已驗證程式。
- 不得以一次大 PR 同時修改資料、UI、導航與產品流程。

## 3. 先證明再重構

任何重構提案必須先列出：

- 現況與實際證據
- 為何不合邏輯或不可維護
- 最小修法
- 資料相容影響
- 回復方式
- 測試方式
- 明確不修改範圍

## 4. 禁止假功能

不得出現：

- 看似可點但沒有作用的篩選器
- 看似可修改但只是靜態文字的設定
- 尚未完成卻未標示的按鈕
- 測試資料冒充正式資料
- 只有畫面沒有儲存或狀態邏輯的入口

功能未完成時，應暫時移除或明確標示尚未開放。

## 5. 直接語言

- 功能名稱必須讓一般使用者一眼理解。
- 禁止重新導入「先放著」「需要你記住的事」「承接責任」等 PMS 抽象文案。
- 技術模型名稱不得直接顯示在使用者畫面。
- 使用者畫面不得顯示內部 ID。

## 6. 資料變更規則

- 新欄位必須提供舊資料安全預設。
- enum 必須處理未知值。
- 任何格式變更前必須建立不可變原始備份。
- 資料異常時必須鎖住寫入。
- 不得刪除或覆蓋無法解析的來源資料。
- 正式遷移必須有筆數、關聯與回復驗證。

## 7. UI 與視覺規則

- UI PR 必須附實際手機尺寸截圖或錄影。
- 不得只以模擬器「能顯示」判定完成。
- 首頁須在 3 秒內說明用途與目前狀態。
- 表單不可像資料庫輸入工具。
- 空狀態必須說明下一步。
- 處理中、等待、逾期、完成與史略要有清楚差異。
- 設計系統先在一頁驗證，經批准後才擴散到其他頁。

## 8. 分支規則

允許的分支前綴：

- `audit/`
- `fix/`
- `feature/`
- `data-migration/`
- `ui/`
- `docs/`
- `ci/`

禁止直接在 `main` 開發。

## 9. PR 必填內容

每個 PR 必須說明：

- 修改目的
- 修改內容
- 修改檔案
- 明確不修改範圍
- 資料影響
- UI 影響
- 測試結果
- 回復方案

## 10. CI 門檻

每個 PR 至少必須通過：

- `flutter analyze`
- `flutter test`
- `flutter build web --release`

資料遷移 PR 另需：

- 舊資料樣本測試
- 部分損壞測試
- 遷移中斷測試
- 重複執行測試
- 回復測試

## 11. 合併規則

不得合併的情況：

- CI 失敗或尚未完成
- PR 超出申報範圍
- 無資料相容說明
- 有未處理高風險問題
- UI 沒有畫面證據
- 違反產品憲法
- 把 PMS 重新帶回生活管理 App

重大架構、資料遷移與正式 UI 方向由專案負責人最終批准。

## 12. Codex 特別規則

Codex 不得：

- 自行改變產品定位
- 自行創造抽象產品哲學
- 自行合併重大 PR
- 為省事整份重寫
- 把多種生活事件硬塞成同一 Task
- 用備註欄冒充正式資料模型
- 未備份就改資料格式
- 看到測試通過就假設 UX 正確

Codex 應：

- 先讀 `docs/control/`
- 遵守最小修改
- 列出不修改範圍
- 以測試與實際差異證明結果
- 發現規格衝突時停止並提出問題，不得自行選邊

## 13. 完成定義

程式能編譯不是完成。只有同時通過產品、資料、UX、視覺與技術驗收，才可標記完成。
