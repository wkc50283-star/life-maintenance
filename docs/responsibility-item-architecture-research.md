## 一、目前已確認的事實

- `Item` 現在的實際角色是可追蹤的項目或責任錨點。使用者透過「先放著」建立資料時，實際寫入的是 `Item`。
- `Schedule` 現在的實際角色是排程與提醒安排；其中 `cardId == 'manual-expiry-reminder'` 的資料同時承擔「需要你記住的事」的事項名稱、提醒日期與取消狀態。
- `Task` 現在的實際角色是由到期 `Schedule` 產生的待處理實例。Today 顯示與完成的是 `Task`，不是直接完成 `Schedule`。
- `MaintenanceRecord` 現在的實際角色是已完成處理後留下的可信履歷紀錄。它可由 Today 完成 Task 後建立，也可由「完成紀錄」入口手動建立。
- v0.12.0 已完成的是手動提醒管理流程，不是完整事項系統。

## 二、目前發現的架構偏移

- 「先放著」目前實際建立 `Item`。
- 「需要你記住的事」目前實際建立 `Schedule`。
- `Schedule.title` 同時承擔事項名稱。
- 取消提醒目前只是將 `Schedule.enabled` 設為 `false`。
- 目前尚無獨立事項本體。

## 三、仍沒有證據的假設

- 使用者是否需要無日期事項。
- 一件事項是否需要多個提醒。
- 使用者是否能理解 `Item`、事項與提醒的差異。
- 現有流程是否已足以滿足主要需求。
- 是否值得承擔資料遷移成本。

## 四、待研究問題

### 無日期事項需求

1. 要驗證什麼：使用者是否需要先記下一件事，但暫時不設定任何日期。
2. 需要什麼證據：實際使用紀錄、訪談、可用性測試，或使用者明確反映「想先記下但不知道日期」。
3. 支持維持現況的結果：多數使用者建立「需要你記住的事」時都能立即填提醒日期，且不覺得日期必填造成阻礙。
4. 支持新增獨立事項 model 的結果：多數使用者需要先建立事項，之後才補日期，且目前必填日期會中斷核心流程。

### 一事項多提醒需求

1. 要驗證什麼：同一件事項是否需要多個提醒安排。
2. 需要什麼證據：使用者案例，例如合約續約需要提前三十天、七天與當天提醒，或同一責任需要多次追蹤。
3. 支持維持現況的結果：一件事項通常只需要一個提醒日期。
4. 支持新增獨立事項 model 的結果：一件事項常見多個提醒，且使用多筆 Schedule 代表同一事項會造成使用者混淆或資料重複。

### 先放著的真實使用情境

1. 要驗證什麼：「先放著」對使用者來說是項目、收件匣、責任錨點，還是一般事項。
2. 需要什麼證據：使用者建立內容的類型、使用者語言、建立後是否會補提醒或紀錄。
3. 支持維持現況的結果：使用者主要把「先放著」當成可長期追蹤的項目或對象。
4. 支持新增獨立事項 model 的結果：使用者主要把「先放著」當成臨時事項收件匣，而不是長期項目。

### 取消提醒與取消事項的理解差異

1. 要驗證什麼：使用者看到「取消提醒」時，理解為取消排程，還是取消整件事項。
2. 需要什麼證據：可用性測試、操作後回訪、使用者對取消後狀態的描述。
3. 支持維持現況的結果：使用者能理解取消後該提醒仍保留在清單並標示已取消。
4. 支持新增獨立事項 model 的結果：使用者期待「取消提醒」與「取消事項」是兩種不同操作。

### 使用者是否會把 Item 當成物品、責任錨點或一般事項

1. 要驗證什麼：`Item` 在使用者心中是否適合作為生活責任的長期錨點。
2. 需要什麼證據：使用者建立的項目名稱、分類使用情況、是否出現非物品型責任。
3. 支持維持現況的結果：使用者能自然把設備、文件、合約、房屋部位等都放進項目中。
4. 支持新增獨立事項 model 的結果：使用者頻繁建立短期事項，且這些事項不適合作為 `Item` 長期存在。

### 舊 manual-expiry-reminder 資料的相容需求

1. 要驗證什麼：若未來新增獨立事項 model，現有 `manual-expiry-reminder` Schedule 是否需要遷移，或只需相容讀取。
2. 需要什麼證據：現有本機資料量、資料欄位完整度、使用者是否仍需要看到舊提醒。
3. 支持維持現況的結果：現有資料量小，且目前 Schedule-based 流程足夠清楚。
4. 支持新增獨立事項 model 的結果：現有資料必須保留歷史語意，且新 model 能安全映射舊資料。

## 五、三種候選方案

### A. 維持現況，只修正文案與定位

- 可解決問題：降低短期開發風險，維持 v0.12.0 手動提醒管理流程穩定，避免資料遷移。
- 無法解決問題：無日期事項、一事項多提醒、事項與排程分離等能力仍無法完整支援。
- 影響層級：主要是文件、文案與小型 UI 狀態修正。
- 舊資料風險：低。
- 開發風險：低。
- 驗證成本：低，可用現有流程與少量使用者測試確認。
- 尚未取得的必要證據：現有 Schedule-based 手動提醒是否已足夠滿足主要需求。

### B. 最小新增獨立事項 model

- 可解決問題：讓事項本體與提醒排程分離，支援無日期事項，未來可支援一事項多提醒。
- 無法解決問題：不能自動解決所有舊資料語意，仍需相容策略與 UI 過渡設計。
- 影響層級：新增 model、repository、讀寫流程、Add/Items/Today 相關 UI 與測試。
- 舊資料風險：中。
- 開發風險：中。
- 驗證成本：中，需要先驗證無日期事項與一事項多提醒是否真的重要。
- 尚未取得的必要證據：使用者確實需要獨立事項本體，且現有架構已阻斷核心需求。

### C. 完整重整事項 / 排程架構

- 可解決問題：重新定義 Item、事項、Schedule、Task、MaintenanceRecord 的完整邊界，長期架構最乾淨。
- 無法解決問題：不能保證短期交付，且可能破壞已完成流程。
- 影響層級：model、repository、service、資料格式、migration、Add/Today/Items/History 主要流程與測試。
- 舊資料風險：高。
- 開發風險：高。
- 驗證成本：高，需要完整產品決策與資料遷移方案。
- 尚未取得的必要證據：目前沒有足夠證據支持立即承擔完整重整成本。

## 六、研究階段決策閘門

只有符合以下條件，才可以向董事長申請進入設計或開發：

- 已有明確使用者證據。
- 現有架構確實阻斷核心需求。
- 舊資料相容方案已完成。
- 最小方案與不改方案已比較。
- 回滾方式已定義。
- 影響檔案、資料格式與測試範圍已列清楚。
- 董事長正式批准。
